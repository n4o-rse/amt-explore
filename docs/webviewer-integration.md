# Connecting the Python pipeline to the existing Webviewer

The webviewer in `docs/` currently has the TTL path hard-coded in
`docs/js/amt.js`:

```js
// Path to the TTL data file (relative to docs/index.htm)
var TTL_PATH = "data/PotterAttributionExample.ttl";
```

To make it accept a file generated by the Python CLI, change those two lines to:

```js
// Path to the TTL data file (relative to docs/index.htm).
// Override via URL parameter:  index.htm?ttl=data/my-export.ttl
var TTL_PATH = (function () {
  var params = new URLSearchParams(window.location.search);
  return params.get("ttl") || "data/PotterAttributionExample.ttl";
})();
```

That's the only change required. After that, the workflow is:

```bash
# 1. Run the Python pipeline – exports a TTL the viewer understands
amt examples/PotterAttributionExample.ttl --reason \
    --export-ttl docs/data/my-export.ttl

# 2. Serve the docs/ folder (any static server)
cd docs && python -m http.server 8000

# 3. Open in browser:
#    http://localhost:8000/index.htm?ttl=data/my-export.ttl
```

## Why this approach

* **No backend, no build step.** The viewer stays a static site you can host
  on GitHub Pages exactly as today.
* **Forward compatible.** Without `?ttl=…` the viewer behaves as before.
* **Reasoning round-trips.** Inferred edges written by `--reason` are
  re-loaded as ordinary assertions in the JS viewer (it currently doesn't
  read the `amt:inferred` flag, but it draws them just like any other edge).
