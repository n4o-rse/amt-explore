# Connecting the Python pipeline to the Webviewer

The webviewer in `docs/` is the JavaScript implementation of the Academic
Meta Tool, ready to be served via GitHub Pages (or any static host).
It accepts TTL files generated by the Python CLI through a URL parameter.

## How it works

`docs/js/amt.js` reads the TTL file path from the `?ttl=…` query parameter,
falling back to the bundled Potter example when no parameter is given:

```js
var TTL_PATH = (function () {
  var params = new URLSearchParams(window.location.search);
  return params.get("ttl") || "data/PotterAttributionExample.ttl";
})();
```

(This is the only change relative to the upstream JS code.)

## End-to-end workflow

```bash
# 1. Run the Python pipeline – exports a TTL the viewer understands.
amt examples/PotterAttributionExample.ttl --reason \
    --export-ttl docs/data/my-export.ttl

# 2. Serve docs/ locally for testing
cd docs && python -m http.server 8000

# 3. Open in browser
#    http://localhost:8000/index.htm?ttl=data/my-export.ttl
```

## GitHub Pages

In the repository settings, enable Pages with **Source: `main` branch,
Folder: `/docs`**. The viewer will then be available at
`https://<user>.github.io/<repo>/`, and you can link to specific datasets
with `…/index.htm?ttl=data/<your-file>.ttl`.

## Notes on round-tripping

* TTL files written by the Python `--reason` exporter contain inferred
  edges (tagged with `amt:inferred "true"^^xsd:boolean`). The current JS
  viewer doesn't read this flag, so inferred edges appear as ordinary
  asserted ones. If you want to keep the visual distinction, export
  *without* `--reason` and let the viewer do its own inference.
* Axioms are preserved across export/import — the JS viewer can re-run
  reasoning on a Python-exported TTL.
